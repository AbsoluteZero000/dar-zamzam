Dar Zamzam Frontend Improvement TODO

This page records the improvement opportunities identified during the repository-wide review. Checkboxes are intentionally left open so the document can serve as an implementation checklist.

Initial review result: the production build and TypeScript validation passed.

Highest priority

  1. Replace tokens stored in localStorage with secure HttpOnly, SameSite cookies; protect routes through middleware or server layouts; load the current user server-side; enforce roles on every backend operation; and add a logout endpoint that invalidates the session.

  2. Keep queued updates until acknowledgement, retry failed saves, show save/error state, flush or warn before navigation, add conflict handling, and consider a batch-attendance endpoint instead of one request per student.

  3. Make request<T> generic, type every endpoint, validate external responses (for example with Zod or generated OpenAPI types), define role/status unions, support cancellation and timeouts, normalize network errors, handle empty 204 responses, and make 401 handling consistent for downloads.

  4. Break up management, attendance, and session-detail pages into feature components, modal/form components, hooks such as useManageData and useSessionAttendance, and shared UI including Modal, StudentAvatar, ImagePreview, and attendance status controls.

  5. Add lint, typecheck, test, and build scripts. Cover filter boolean logic, week/date boundaries, 401 and API errors, autosave retry behavior, forms, login, attendance editing, session confirmation, and admin authorization. Run these checks in CI.

Performance and data flow

  1. Add consolidated dashboard and management endpoints, include students with sheikh queries where appropriate, start independent calls concurrently, and consider TanStack Query or SWR for caching, deduplication, retry, and invalidation.

  2. Remove the duplicate initial grid fetch and abort earlier requests when the selected sheikh or filters change so an older response cannot overwrite current state.

  3. Stop swallowing request failures or displaying zero-valued statistics as real data. Provide shared error UI, meaningful messages, retry actions, and reliable finally handling so pages cannot remain stuck loading.

Architecture and Next.js

  1. Move metadata and the document shell to a server layout, isolate navigation/theme interactivity in a client AppShell, introduce public/authenticated route groups, and use dedicated layouts for landing, login, and authenticated application routes.

  2. Create error.tsx, not-found.tsx, and appropriate loading.tsx files, including an authenticated-area error boundary where useful.

  3. Target a modern runtime such as ES2020, consider disabling allowJs, enable stricter options such as noUncheckedIndexedAccess and exactOptionalPropertyTypes after model cleanup, explicitly exclude generated folders, and stop tracking tsconfig.tsbuildinfo.

Accessibility and user experience

  1. Add dialog roles, accessible titles, focus trapping and restoration, Escape handling, scroll locking, proper htmlFor/id associations, and aria-label text for icon-only controls.

  2. Use shared confirmation dialogs and toast/notice components for consistent styling, accessibility, and operation-specific feedback.

  3. Remove the global window.__dragData state, use component state or an accessible maintained library, and provide keyboard, touch, explicit move, and reorder alternatives.

  4. Create smaller WebM/MP4 variants and a poster, avoid unnecessary video preload, honor reduced-motion and data-saving preferences, lazy-load noncritical media, and configure next/image for profile images where practical.

Deployment and maintenance

  1. Replace the hard-coded Docker build value with an explicit build argument or same-origin proxy. A same-origin design can also simplify cookies and CORS.

  2. Review whether the standalone output needs the full production node_modules copy, use npm ci --omit=dev if retained, run as a non-root user, add health checks, pin the base image for reproducibility, test the 256 MB memory allocation, and document rollback procedures.

  3. Create a README covering local setup, environment variables, backend/API requirements, authentication, roles and workflows, Arabic/RTL and date conventions, testing, deployment, and database backup/export expectations.

Recommended implementation order

  1. Fix autosave reliability.
  2. Secure authentication and session handling.
  3. Type and validate the API client.
  4. Add tests for attendance, authorization, and dates.
  5. Split management, attendance, and session-detail features.
  6. Consolidate backend requests and introduce caching.
  7. Improve error states and accessibility.
  8. Optimize landing media and deployment configuration.
