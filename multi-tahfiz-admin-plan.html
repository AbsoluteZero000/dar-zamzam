Why the current model cannot express this

The code currently stores a single users.tahfiz_id. A normal admin is therefore bound to exactly one Tahfiz. A platform super admin has no bound Tahfiz and can enter any branch using the support context.

Keep

Tenant-scoped records already carry tahfiz_id, and backend queries consistently filter through a tenant context. This is a solid isolation boundary.

Change

Replace the one-user-to-one-Tahfiz relationship with explicit memberships. Make the active Tahfiz a request context, not a permanent property of the login.

Avoid

Do not turn this account into super_admin, and do not add branch IDs directly into a long-lived JWT. Both approaches create avoidable privilege and revocation risks.

Recommended data model

A join table makes the permission boundary explicit and supports different roles in different branches later.

Authentication and request flow

The JWT identifies the user. The database decides which Tahfiz that user may access on every tenant-scoped request.

Token contains immutable user ID and global role only.

→
3 · AuthorizeValidate membership
Backend loads an active membership and derives the tenant role.
A changed or revoked membership takes effect immediately because authorization is checked from the database, not trusted from an old token or browser storage.

Proposed user experience

After login, open the only branch automatically. If the user has multiple memberships, show a branch chooser and keep a visible switcher in the navigation bar.

Interaction rules

  • Always display the current branch name in desktop and mobile navigation.
  • Changing branches clears branch-specific cached state, filters, and in-progress forms.
  • Store only the last selected accessible Tahfiz ID in local storage; the server remains authoritative.
  • If access is revoked, show “You no longer have access” and return to the chooser.
  • Use “My branches” for scoped users; reserve the existing “Platform” screen for global super admins.

Security invariants

These rules are more important than the switcher UI. They prevent a user from changing a header or URL to enter another Tahfiz.

Required

  • Every tenant endpoint resolves an active membership for (user_id, X-Tahfiz-ID).
  • Admin mutations require that membership’s role to be admin.
  • A sheikh_id on a membership must belong to the same Tahfiz.
  • Audit branch switches, membership grants, role changes, and revocations.
  • Only a platform super admin can grant one account access across unrelated Tahfiz branches in the first release.

Never trust

  • A Tahfiz ID from local storage without server validation.
  • The legacy users.tahfiz_id after the migration is complete.
  • A membership or role list embedded in a long-lived token.
  • A frontend-only restriction hiding branches from the menu.
  • A selected sheikh that has not been checked against the active Tahfiz.

Backend and frontend changes

The existing support header can be generalized, but its authorization behavior must differ between scoped members and platform super admins.

Backward-compatible migration

No current account should lose access, and this should not require resetting passwords.

1

Add without removing

Create the membership table and default_tahfiz_id. Leave users.tahfiz_id and current role behavior intact.

2

Backfill memberships

For every non-super-admin with a Tahfiz ID, insert one membership using the existing role and sheikh assignment. Copy its Tahfiz into the default field.

3

Dual-read, membership-first

Authorize from memberships when present; temporarily fall back to the legacy field for unmigrated rows. Log every fallback.

4

Enable the two-branch account

Create or select the shared user, then grant admin memberships in Faisal and Henedy. Verify isolation against a third Tahfiz.

5

Remove legacy authorization

After one stable release and a database backup, stop reading users.tahfiz_id for access decisions. Column removal can wait for a later maintenance release.

Acceptance and security tests

The release is complete only when cross-tenant isolation is proven, not merely when the branch selector works.

Happy paths

  • One login lists exactly Faisal and Henedy.
  • Switching changes dashboards, students, sheikhs, sessions, reports, settings, and exports.
  • The account can perform admin actions in both assigned branches.
  • A normal single-branch user still enters its branch directly.
  • Platform super-admin support access still works and remains audited.

Denial paths

  • Changing X-Tahfiz-ID to a third branch returns 403.
  • A sheikh membership cannot call tenant-admin endpoints.
  • A revoked membership stops working immediately with the same JWT.
  • A suspended Tahfiz remains inaccessible even with a membership.
  • IDs from one branch cannot be used in mutations while another branch is active.

Suggested delivery plan

Ship the authorization foundation before the visual selector so security behavior is testable in isolation.

Phase A

Foundation

Schema, backfill, membership-aware tenant context, audit events, and backend contract tests.

Phase B

Product experience

Membership APIs, login routing, branch chooser, navigation switcher, and cache reset behavior.

Phase C

Controlled activation

Grant the shared account access to Faisal and Henedy, run isolation checks, then monitor audit logs.

Final decision

Build now
Membership model + branch switcher + super-admin-managed grants.
Defer for later
Self-service organizations, custom permission matrices, and cross-branch combined reporting.
